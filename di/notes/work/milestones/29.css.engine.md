
# Is it really too ad-hoc?

Of all the options available in css, the app uses a lot. Most of them hard wired with some number that made sense at the moment. now, quite a bit of code later, it smells to me -- too ad hoc, fragile, missing a mental model. it is time to heft our maintenance cost, going forward.

---

## proposal

**goal:** tighten the pipeline. don't rebuild — clean up the seams.

1. **audit magic numbers** — grep for hardcoded px values in `<style>` blocks. each one either gets pulled into Constants.ts or justified with a comment. this is the biggest source of "jerry-rigged" feel.

2. **close the color leaks** — every color should flow through Colors.ts → CSS vars. find hardcoded color strings in components and route them through the system. small effort, big consistency win.

3. **clarify the inline-vs-var boundary** — rule of thumb: if a value changes at runtime (layout dimensions, positions), inline style:. if it's a design token (gaps, corners, colors), CSS var. right now some tokens sneak into inline styles. move them to vars.

4. **pseudo-element review** — look at each ::before/::after usage. some are genuine (gradient overlays for hideable sections). others might be replaceable with simpler CSS or a wrapper element. decide case by case.

5. **document the contract** — add a short comment block at the top of Constants.ts and Colors.ts explaining what they provide and how components should consume them. the system is actually well-designed — it just doesn't announce itself.

**order:** 1 → 2 → 3 → 4 → 5. each step is independent and small. can be done across multiple sessions.

## leaks

**colors — biggest gap.**
`white` appears 30+ times, `black` 20+ times, `cornflowerblue` 5+ times, `#007aff` in 3+ files, `#ccc` and `darkgray` scattered. none flow through Colors.ts or CSS vars.

**borders — `0.5px solid currentColor`**
appears ~15 times. same pattern, never named.

**focus outline — `1.5px solid cornflowerblue`**
appears in 5+ files. copy-pasted.

**slider styling**
`#007aff` (thumb), `#ccc` (track), `darkgray` (border) repeated across Slider, P_Angles, P_Repeat, D_Preferences.

**small spacers**
hardcoded `3px`, `4px`, `5px`, `6px` divs and margins everywhere. micro-adjustments that mostly derive from `layout.gap` but don't use it.

**graph overlays**
font sizes (`10px`, `12px`), input widths (`60px`, `80px`), position offsets (`10px`, `14px`) all hardcoded in Graph.svelte. self-contained but brittle.

## architecture

### three-tier hybrid

TS constants → CSS custom properties → component styles

```text
Constants.ts (design tokens: all measurements derived from common_size)
    ↓
App.svelte onMount (injects --l-gap, --h-controls, --corner-common, --text, --accent, --bg onto :root)
    ↓
Component-level scoped <style> blocks (use var(--x) for theming/spacing)
    + inline style: directives (style:width, style:height — for reactive computed dimensions)
    + class: bindings (class:active — for state-driven presentation)
    + data-hit attributes (set by Hit_Target.ts — CSS responds with attribute selectors)
```

**layout:** JS state (window size + visibility stores) → $derived values → inline styles
**colors:** Colors.ts stores → $effect in App.svelte → CSS vars → scoped CSS
**hover:** hit detection system → data-hit attribute → CSS attribute selectors
**spacing/corners/z-index:** Constants.ts → CSS vars → scoped CSS

no external .css files. everything scoped inside .svelte components.

## context

**is this a typical software architecture?**
yes, pretty typical for this kind of app. having a TS runtime as source of truth that feeds CSS is the standard pattern in Svelte/React/Vue apps that need layout responsive to application state (not just viewport). design systems like Chakra, Radix, and Tailwind's JS config all do variants of this. the three-tier flow (tokens → CSS vars → scoped styles) is essentially what design-system tooling formalizes.

what's less typical is doing it without a framework/library layer — Constants.ts + Colors.ts + manual `root.setProperty()` instead of a CSS-in-JS library or design token pipeline. leaner, but discipline has to come from convention rather than tooling.

**is it very CSS heavy?**
no. it's TS-heavy. CSS is the output layer — the engine is JS/TS stores and derived state. CSS mostly consumes variables, it doesn't define the logic. the real work happens in Constants.ts, Colors.ts, and $derived computations in Main.svelte.

**does it need a rethink?**
not a full rethink. the bones are good: centralized tokens, programmatic color derivation, reactive injection. but it has accumulated friction: magic numbers leak in, some colors bypass Colors.ts, inline styles and CSS vars overlap. the proposal above addresses all of it incrementally.
