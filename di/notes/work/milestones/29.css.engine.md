## answers
### **is this a typical software architecture?**
focusing on runtime SOT.

Yes, pretty typical for this kind of app. Having a TS runtime as source of truth that feeds CSS is the standard pattern in Svelte/React/Vue apps that need layout responsive to application state (not just viewport). Design systems like Chakra, Radix, and Tailwind's JS config all do variants of this. The three-tier flow (tokens â†’ CSS vars â†’ scoped styles) is essentially what design-system tooling formalizes.

What's _less_ typical is doing it without a framework/library layer â€” you rolled your own Constants.ts + Colors.ts + manual `root.setProperty()` injection instead of using something like a CSS-in-JS library or a design token pipeline tool. That's not wrong â€” it's leaner â€” but it means the discipline has to come from convention rather than tooling, which is where the ad-hoc feeling creeps in.

### **is it very css heavy?**
no. it's TS-heavy. CSS is the _output_ layer â€” the engine is JS/TS stores and derived state. CSS mostly consumes variables, it doesn't define the logic. the real work happens in Constants.ts, Colors.ts, and $derived computations in Main.svelte.

### **is this a good thing?**
mixed. the TS-driven approach means layout responds to state correctly (details panel open/close, window resize). pure CSS can't react to JS state without JS bridging anyway. but it means you have two parallel systems â€” TS computing dimensions and CSS consuming them â€” that can drift apart.

### **does it need a rethink?**
not a full rethink. the bones are good: centralized tokens, programmatic color derivation, reactive injection. but it has accumulated friction:

- magic numbers leak in (padding: 3px, margin: 4px in several components)
- some colors bypass the Colors system (hardcoded 'darkgray', rgba(0,0,0,0.5))
- pseudo-element tricks (Separator::before, Hideable::before) feel like workarounds
- inline styles and CSS vars overlap â€” sometimes the same value flows through both paths

## proposal

**goal:** tighten the pipeline. don't rebuild â€” clean up the seams.

1. **audit magic numbers** â€” grep for hardcoded px values in `<style>` blocks. each one either gets pulled into Constants.ts or justified with a comment. this is the biggest source of "jerry-rigged" feel.

2. **close the color leaks** â€” every color should flow through Colors.ts â†’ CSS vars. find hardcoded color strings in components and route them through the system. small effort, big consistency win.

3. **clarify the inline-vs-var boundary** â€” rule of thumb: if a value changes at runtime (layout dimensions, positions), inline style:. if it's a design token (gaps, corners, colors), CSS var. right now some tokens sneak into inline styles. move them to vars.

4. **pseudo-element review** â€” look at each ::before/::after usage. some are genuine (gradient overlays for hideable sections). others might be replaceable with simpler CSS or a wrapper element. decide case by case.

5. **document the contract** â€” add a short comment block at the top of Constants.ts and Colors.ts explaining what they provide and how components should consume them. the system is actually well-designed â€” it just doesn't announce itself.

**order:** 1 â†’ 2 â†’ 3 â†’ 4 â†’ 5. each step is independent and small. can be done across multiple sessions.
