### DONE step 3.1 — Persistence between renders. Tolerance = 5 pixels (2026-06-11)

Encoded as rule 2a of [dim.spec.md](dimensionals%20spec.md), in section A.

Done across four slices, each leaving the running app working and each its own commit-worthy unit:

- **DONE slice 3.1A — record state.** At the end of every successful render, write the chosen edge / face / witness index / label position fraction per (part, axis) into a module-level persistence map. Prune entries for parts no longer rendered. Reads happen in later slices. Two unit tests cover the record-on-winner and prune-on-disappear behaviour.
- **DONE slice 3.1B — skip path.** After the witness-index vote and before the main search loop, re-project every persisted entry to the current frame and run the four viability checks (winners list, label position within range plus 5-pixel slack, pair clearance, silhouette non-intersection). If all entries pass AND the eligible (part, axis) set has not changed, build the result from the re-projected entries and return early; otherwise fall through to the full search. Two unit tests cover the same-scene-skips and changed-scene-bypasses cases.
- **DONE slice 3.1C — seeded run.** When the skip path fails, every persisted record that passes the STRICT (no-tolerance) versions of the same four checks gets LOCKED — its rectangle, anchors, witness lines, and dim line go into the obstacle lists before the main loop runs, and its placement record goes straight into the result. The main loop skips those (part, axis) pairs, so the free dimensions search around the locked ones. One unit test covers locking-and-persistence-preserved when a new part joins the scene.
- **DONE slice 3.1D — drift safety.** A counter tracks consecutive skip-renders in which any label's position passed only by the 5-pixel tolerance (would have failed the strict [0, 1] check). After two such skips, the next render forces a full search to flush accumulated drift. The counter resets after a clean skip or after any full search. One unit test covers the clean-skip case — counter stays at zero when nothing drifts.

Six unit tests in total. svelte-check clean, 824 unit tests pass.

Each label's previously chosen values — which edge, which face, which witness index, which label position — are remembered between renders. On the next render, four viability checks per label decide whether the full search can be skipped:

- The label's previous witness index is still on the post-vote winners list for its face.
- The label's previous position along the dim line lies within (new range start − 5 px) to (new range end + 5 px).
- The label's rectangle, at its remembered position, still clears every other label's remembered rectangle by their pair-clearance value.
- The label's rectangle, at its remembered position, does not cross inside the silhouette polygon (touching from outside is fine — matches the main search path's silhouette filter).

If every label passes all four checks, the full search is skipped; the remembered values are reused for this render. If any label fails any check, the full search runs — seeded with last render's values. Inside the seeded run, labels that pass the strict (no-tolerance) checks are LOCKED in place and act as fixed obstacles; only labels that failed get re-placed.

**Drift safety.** After two consecutive skip-renders in which any check passed only by the 5-pixel tolerance (would have failed without it), force a full search on the next render anyway. Stops slow drift.

**Tuning note.** The 5-pixel value is a starting point. Worth measuring against real tumble sessions before treating it as locked in.
